CPU e Linguagem C

🔵 CPU e Linguagem C — parte essencial da jornada para compreender o funcionamento interno do processador.

⚙️ Como o Compilador Funciona

Quando você compila um programa em C, uma série de transformações acontecem. Cada etapa traduz seu código para algo mais próximo da máquina:

🔧 Etapas da Compilação
📝 Código Fonte (.c)
int main() { printf("Hello"); }
↓ Pré-processador (cpp)
📄 Código Expandido
Headers incluídos, macros expandidas, comentários removidos
↓ Compilador (gcc -S)
🔧 Assembly (.s)
mov eax, 1 | call printf | ret
↓ Assembler (as)
💻 Código Objeto (.o)
Código de máquina binário + tabela de símbolos
↓ Linker (ld)
🚀 Executável
Programa completo pronto para execução 
// Você pode ver cada etapa:
$ gcc -E programa.c > programa.i    // Pré-processado
$ gcc -S programa.c                 // Gera programa.s (Assembly)
$ gcc -c programa.c                 // Gera programa.o (Objeto)
$ gcc programa.c -o programa        // Executável completo

// Ver o Assembly gerado:
$ objdump -d programa | less
// Ou:
$ gcc -O0 -S programa.c && cat programa.s
💡 Dica Prática

Use gcc -O0 para compilar SEM otimizações — o Assembly gerado será mais fácil de entender e mapear para seu código C. Com -O3, o compilador reorganiza tudo para ser mais rápido, mas o Assembly fica confuso.

📦 Variáveis na CPU

Cada tipo de variável em C tem uma representação diferente no nível da CPU:

// Código C:
int x = 42;           // 4 bytes (32 bits), complemento de 2
char c = 'A';         // 1 byte, ASCII = 65
float f = 3.14f;      // 4 bytes, padrão IEEE 754
double d = 2.718;     // 8 bytes, ponto flutuante duplo
int *ptr = &x;        // 8 bytes (64-bit), endereço de memória

// Como a CPU vê isso (Assembly x86_64):
// mov DWORD PTR [rbp-4], 42     // x = 42 (DWORD = 4 bytes)
// mov BYTE PTR [rbp-5], 65      // c = 'A' (BYTE = 1 byte)
// movss xmm0, [rip+.LC0]        // f = 3.14 (xmm = registrador FPU)
// mov QWORD PTR [rbp-24], rax   // ptr = endereço (QWORD = 8 bytes)

// Tamanhos em C:
// sizeof(char)      = 1 byte
// sizeof(short)     = 2 bytes
// sizeof(int)       = 4 bytes
// sizeof(long)      = 8 bytes (Linux 64-bit)
// sizeof(long long) = 8 bytes
// sizeof(float)     = 4 bytes
// sizeof(double)    = 8 bytes
// sizeof(void*)     = 8 bytes (64-bit)
📊 Representação na Memória
int x = 42
0x2A 0x00 0x00 0x00
(little-endian: byte menos significativo primeiro)
char c = 'A'
0x41
(ASCII 65 = 'A')
float f = 3.14f
0xC3 0xF5 0x48 0x40
(IEEE 754 single precision)
int *ptr
0x34 0x12 0x7F 0x00...
(8 bytes, endereço de memória)

🔄 Loops na CPU

Loops em C são traduzidos para saltos condicionais em Assembly:

// C:
for (int i = 0; i < 10; i++) {
    printf("%d
", i);
}

// Assembly x86_64 (simplificado):
// mov DWORD PTR [rbp-4], 0    // i = 0
// .L_loop:
// cmp DWORD PTR [rbp-4], 10   // compara i com 10
// jge .L_end                  // se i >= 10, pula para fim
// 
// // corpo do loop
// mov eax, [rbp-4]            // eax = i
// call printf                 // imprime
// 
// add DWORD PTR [rbp-4], 1    // i++
// jmp .L_loop                 // volta para o início
// .L_end:

// O loop é implementado com:
// - Inicialização (mov)
// - Comparação (cmp)
// - Salto condicional (jge = jump if greater or equal)
// - Incremento (add)
// - Salto incondicional (jmp)
🤔 Curiosidade

O compilador pode "desenrolar" (unroll) loops automaticamente. Em vez de 10 iterações com saltos, ele pode gerar 10 instruções sequenciais — eliminando o overhead dos saltos e melhorando o uso do cache!

📞 Funções na CPU

Chamadas de função envolvem a stack, registradores e convenções de chamada:

// C:
int soma(int a, int b) {
    return a + b;
}

int main() {
    int resultado = soma(10, 20);
    return 0;
}

// Assembly x86_64 (System V AMD64 ABI - Linux):
// main:
//   mov edi, 10          // 1º argumento em EDI (a)
//   mov esi, 20          // 2º argumento em ESI (b)
//   call soma            // chama função (push RIP, jump)
//   mov [rbp-4], eax     // resultado = valor retornado em EAX
//   mov eax, 0           // return 0
//   ret
//
// soma:
//   push rbp             // salva frame anterior
//   mov rbp, rsp         // novo frame
//   mov eax, edi         // eax = a (1º arg)
//   add eax, esi         // eax += b (2º arg)
//   pop rbp              // restaura frame
//   ret                  // retorna (eax = resultado)

// Convenção de chamada Linux x86_64:
// Argumentos inteiros: RDI, RSI, RDX, RCX, R8, R9
// Retorno: RAX
// Preservados pelo chamado: RBX, RBP, R12-R15
// Voláteis: RAX, RCX, RDX, RSI, RDI, R8-R11
💡 Convenção de Chamada

A "convenção de chamada" define COMO passar argumentos, QUEM limpa a stack, e ONDE colocar o retorno. Diferentes sistemas operacionais e arquiteturas têm convenções diferentes! Linux x86_64 usa a ABI System V, enquanto Windows usa a convenção Microsoft x64.

🔗 Ponteiros na CPU

Ponteiros são endereços de memória. A CPU os trata como números inteiros, mas com semântica especial:

// C:
int x = 42;
int *ptr = &x;
int valor = *ptr;     // dereference: lê valor no endereço
ptr++;                // aritmética de ponteiros

// Assembly:
// lea rax, [rbp-4]         // rax = &x (endereço de x)
// mov QWORD PTR [rbp-16], rax  // ptr = rax
// mov rax, [rbp-16]        // rax = ptr (endereço)
// mov eax, [rax]           // eax = *ptr (valor no endereço)
// mov [rbp-20], eax        // valor = eax
//
// ptr++:
// mov rax, [rbp-16]        // rax = ptr
// add rax, 4               // rax += 4 (sizeof(int))
// mov [rbp-16], rax        // ptr = rax

// Aritmética de ponteiros:
// ptr + 1  →  endereço + sizeof(tipo)
// int*  + 1 = +4 bytes
// char* + 1 = +1 byte
// double* + 1 = +8 bytes
⚠️ Segmentation Fault

Quando você dereferencia um ponteiro inválido (NULL, endereço não mapeado), a CPU gera uma exceção que o kernel captura e mata seu programa com "Segmentation fault". É o kernel protegendo a memória!
