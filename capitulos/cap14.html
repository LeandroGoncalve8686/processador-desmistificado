🔍 Engenharia Reversa

Engenharia reversa é o processo de analisar um programa compilado para entender seu funcionamento — sem ter o código fonte:

// Ferramentas de engenharia reversa:
// • objdump / readelf   — análise estática de binários Linux
// • gdb                 — debugger (executa passo a passo)
// • radare2 / Cutter    — framework de engenharia reversa
// • Ghidra / IDA Pro    — desassemblers avançados

// Exemplo: descobrir o que um programa faz
$ objdump -d programa | grep -A5 main
// Mostra o Assembly da função main

// No gdb:
$ gdb ./programa
(gdb) break main        // para no início de main
(gdb) run               // executa
(gdb) info registers    // mostra todos os registradores
(gdb) x/10i $rip        // mostra 10 instruções no RIP
(gdb) stepi             // executa 1 instrução
💡 Aplicação Defensiva

Analistas de malware usam engenharia reversa para entender como vírus funcionam e criar assinaturas de detecção. Sem entender Assembly e registradores, é impossível analisar malware moderno.

💥 Buffer Overflow — Conceito

Buffer overflow ocorre quando um programa escreve MAIS dados do que um buffer pode conter, sobrescrevendo áreas adjacentes da memória:

// Código VULNERÁVEL (NÃO FAÇA ISSO!):
void func_vulneravel() {
    char buffer[16];           // Buffer de 16 bytes na stack
    gets(buffer);              // gets NÃO verifica tamanho!
    // Se usuário digitar mais de 16 caracteres...
    // ...os bytes extras sobrescrevem a stack!
}

// Layout da stack ANTES do overflow:
// [buffer 16 bytes] [RBP salvo 8 bytes] [Retorno 8 bytes]
//
// Se digitar 24 'A's:
// [AAAAAAAAAAAAAAAA] [AAAAAAAA] [0x4141414141414141]
// buffer preenchido   RBP        ENDEREÇO DE RETORNO!
//
// Quando a função retorna, RIP = 0x4141414141414141
// Isso causa um crash... ou pior, executa código malicioso!

// DEFESA: use funções seguras!
// fgets(buffer, sizeof(buffer), stdin);  // limita a 16 bytes
📊 Stack Antes e Depois do Overflow

✅ Normal

↑ Endereços altos
End. Retorno: 0x401234
RBP salvo: 0x7FFF...
Buffer[16]: "Hello"
↓ Endereços baixos

❌ Overflow

🛡️ Defesas Modernas

Sistemas modernos têm múltiplas camadas de defesa contra exploits:

🛡️ Camadas de Proteção
🥞 Stack Canaries
Valor aleatório colocado entre buffer e endereço de retorno. Se for alterado, o programa aborta antes de retornar.
🎲 ASLR (Address Space Layout Randomization)
Randomiza endereços de stack, heap e bibliotecas a cada execução. Dificulta saber para onde pular.
🚫 DEP/NX (Data Execution Prevention)
Stack e heap são marcados como NÃO EXECUTÁVEIS. Código injetado na stack não pode ser executado.
🔗 RELRO (Relocation Read-Only)
Torna a tabela de relocations somente leitura após carregamento, impedindo sobrescrita de endereços de funções. 
// Verificar proteções de um binário Linux:
$ checksec --file=./programa
// RELRO:    Full RELRO
// Stack:    Canary found
// NX:       NX enabled
// PIE:      PIE enabled
// ASLR:     ASLR enabled

// checksec mostra quais defesas estão ativas
// Quanto mais "verde", mais seguro o binário

🔗 ROP (Return-Oriented Programming)

Com DEP/NX ativo, não podemos executar código na stack. Mas podemos REUTILIZAR pedaços de código já existentes no programa (gadgets):

// Conceito de ROP (simplificado):
// Em vez de injetar código, o atacante:
// 1. Sobrescreve o endereço de retorno
// 2. Aponta para "gadgets" — pequenas sequências de instruções
//    que já existem no programa ou bibliotecas
// 3. Cadeia múltiplos gadgets para fazer o que quiser

// Exemplo de gadget:
// pop rax; ret        // Tira valor da stack e coloca em RAX
// mov rdi, rax; ret   // Move RAX para RDI (1º argumento)
// syscall; ret        // Executa syscall

// Cadeia ROP para chamar execve("/bin/sh"):
// [pop rax; ret] [59]           // rax = 59 (syscall execve)
// [pop rdi; ret] [addr /bin/sh] // rdi = ponteiro para "/bin/sh"
// [pop rsi; ret] [0]            // rsi = NULL
// [pop rdx; ret] [0]            // rdx = NULL
// [syscall; ret]                // execve("/bin/sh", NULL, NULL)

// DEFESA: CFI (Control-Flow Integrity)
// Verifica se saltos são para locais válidos
💡 Por que estudar isso?

Entender ROP e defesas é essencial para: 1) Desenvolver software seguro, 2) Analisar vulnerabilidades, 3) Criar ferramentas de detecção, 4) Auditar código, 5) Proteger infraestrutura crítica.
