CPU e Sistema Operacional

🔵 CPU e Sistema Operacional — parte essencial da jornada para compreender o funcionamento interno do processador.

🐧 Como o Linux Conversa com a CPU

O Linux é um sistema operacional que gerencia recursos de hardware, incluindo a CPU. Ele decide qual programa executa, por quanto tempo, e como compartilhar recursos entre processos.

💡 Analogia

O Linux é como um maestro de orquestra. A CPU é a orquestra. O maestro decide qual instrumento (processo) toca quando, por quanto tempo, e como todos se sincronizam para criar uma sinfonia harmoniosa.

📋 Processos e Threads no Linux

No Linux, tudo é representado como um processo:

// Em C, você pode ver seu próprio PID (Process ID):
#include 
#include 

int main() {
    printf("Meu PID: %d
", getpid());
    printf("PID do meu pai: %d
", getppid());

    // Criar um novo processo:
    pid_t filho = fork();

    if (filho == 0) {
        printf("Sou o processo FILHO! PID=%d
", getpid());
    } else {
        printf("Sou o processo PAI! PID=%d, filho=%d
", getpid(), filho);
    }
    return 0;
}

// Cada processo tem:
// - PID único
// - Espaço de memória próprio
// - Tabela de arquivos abertos
// - Contexto de execução (registradores)
🧬 Processo vs Thread
📋 Processo
✓ Espaço de memória próprio
✓ PID único
✓ Tabela de arquivos própria
✓ Independente (crash não afeta outros)
✗ Criação lenta (fork)
✗ Comunicação complexa (IPC)
🧵 Thread
✓ Compartilha memória do processo
✓ TID único (dentro do processo)
✓ Mesma tabela de arquivos
✗ Crash afeta todo o processo
✓ Criação rápida (clone)
✓ Comunicação direta (memória compartilhada)
🤔 Curiosidade

No Linux, threads são implementadas como processos leves (lightweight processes) usando a chamada de sistema clone(). Isso é diferente de outros sistemas operacionais! Você pode ver isso com ps -eLf que mostra threads como processos separados.

⏱️ Escalonamento (Scheduling)

O escalonador do Linux decide QUAL processo/thread executa em CADA núcleo e POR QUANTO TEMPO:

// O escalonador usa o algoritmo CFS (Completely Fair Scheduler):
// - Cada processo recebe uma "fatia de tempo"
// - Processos interativos (GUI) têm prioridade
// - Processos em background usam CPU ociosa
// - O timer interrompe a CPU para trocar processos

// Você pode ver o escalonador em ação:
$ top
// Mostra processos ordenados por uso de CPU
// Cada linha é um processo competindo por tempo de CPU

$ ps -eo pid,comm,psr,%cpu | head
// PSR = qual núcleo está executando
// %CPU = percentual de uso

🔄 Troca de Contexto (Context Switch)

Quando o escalonador decide trocar de processo, a CPU precisa salvar o estado do processo atual e carregar o estado do próximo:

// Troca de contexto envolve:
1. Timer interrompe a CPU (interrupção)
2. Kernel decide qual processo executa agora
3. Salva registradores do processo atual na stack
4. Salva RIP, RSP, RFLAGS na TCB (Task Control Block)
5. Carrega registradores do próximo processo
6. Atualiza tabelas de memória (MMU)
7. Retorna ao User Mode com novo processo

// Custo: ~1-10 microssegundos
// Parece pouco, mas milhares de trocas/segundo somam!

// Ver trocas de contexto:
$ vmstat 1
// A coluna 'cs' mostra context switches por segundo
⚠️ Custo da Troca de Contexto

Cada troca de contexto invalida caches (cache misses ao carregar novo processo) e consiga tempo do kernel. Programas com muitas threads podem passar mais tempo trocando contexto do que executando útil!
