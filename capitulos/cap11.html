Modos de Execução

🔴 Modos de Execução — parte essencial da jornada para compreender o funcionamento interno do processador.

🛡️ User Mode vs Kernel Mode

Processadores modernos operam em dois modos distintos de execução, cada um com privilégios diferentes. Essa separação é a base da segurança dos sistemas operacionais.

🆚 User Mode vs Kernel Mode
👤 User Mode (Ring 3)
✓ Programas de usuário rodam aqui
✓ Acesso limitado à memória
✗ NÃO pode acessar hardware diretamente
✗ NÃO pode modificar tabelas de memória
✗ NÃO pode desabilitar interrupções
✓ Se crashar, não derruba o sistema
👑 Kernel Mode (Ring 0)
✓ Sistema operacional roda aqui
✓ Acesso TOTAL à memória
✓ Pode acessar hardware diretamente
✓ Pode modificar tabelas de memória
✓ Pode desabilitar interrupções
⚠ Se crashar, o SISTEMA inteiro cai
💡 Analogia

User Mode é como um visitante em um museu — pode olhar, mas não tocar. Kernel Mode é o curador — tem a chave de tudo e pode mexer em qualquer exposição.

🔄 Transição entre Modos

Quando um programa precisa fazer algo privilegiado (como ler um arquivo), ele faz uma syscall (chamada de sistema):

// Em C, quando você faz:
FILE *f = fopen("dados.txt", "r");
// O programa está em USER MODE

// Internamente, a biblioteca C faz uma SYSCALL:
// mov rax, 2        // syscall number 2 = open
// mov rdi, filename // nome do arquivo
// mov rsi, flags    // flags de abertura
// syscall           // INSTRUÇÃO ESPECIAL!

// A instrução SYSCALL:
// 1. Troca para KERNEL MODE
// 2. Salva estado do programa
// 3. Executa código do kernel
// 4. Retorna resultado
// 5. Volta para USER MODE

// O kernel verifica permissões e acessa o disco
// Se tudo ok, retorna um file descriptor
// Se não tem permissão, retorna erro (-1)
🔄 Fluxo de uma Syscall
👤 Programa em User Mode
Quer ler um arquivo
↓ syscall
⚡ CPU troca para Kernel Mode
Salva estado, verifica permissões
↓
👑 Kernel executa operação
Acessa disco, verifica segurança
↓
🔄 Retorna resultado
Volta para User Mode

🐧 Relação com Linux e Windows

Ambos os sistemas operacionais usam a mesma arquitetura de proteção, mas com diferenças:

🖥️ Linux vs Windows — Arquitetura de Proteção
🐧 Linux
• Syscalls: números (ex: 0=read, 1=write, 2=open)
• Kernel monolítico (tudo no kernel)
• /proc e /sys expõem kernel ao usuário
• strace mostra syscalls em tempo real
• Código aberto, kernel visível
🪟 Windows
• Syscalls: números diferentes do Linux
• Kernel híbrido (monolítico + microkernel)
• APIs de usuário (Win32, NTAPI)
• Process Monitor mostra operações
• Kernel fechado, APIs documentadas 
// Em Linux, você pode ver as syscalls de um programa:
// Terminal:
$ strace ./meu_programa
// Mostra TODAS as chamadas de sistema em tempo real!

// Exemplo de saída:
// openat(AT_FDCWD, "dados.txt", O_RDONLY) = 3
// read(3, "conteudo...", 1024) = 100
// close(3) = 0

// Cada linha é uma transição User→Kernel→User!
🤔 Curiosidade

Em Linux, tudo é um arquivo! Dispositivos de hardware, processos, informações do sistema — tudo aparece como arquivos em /dev, /proc e /sys. Quando você lê /proc/cpuinfo, o kernel gera as informações da CPU em tempo real!
